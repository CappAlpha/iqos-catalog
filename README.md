# Приложение для подключения IQOS DUO (web, mobile для Android)

Приложение реализует взаимодействие с API, подключение к устройству IQOS DUO по Bluetooth и USB с возможностью запуска в браузере (с использованием Web API для Bluetooth и USB) и в мобильном приложении для Android (с использованием нативных функций).

Подключение производилось на устройстве `IQOS 3.1 (DUO) SW Charger v21.2.0` c такими параметрами:
```
{
  usb: {
    vendorId: 10073,
    productId: 3,
  },
  bluetooth: {
    serviceInfo: '0000180a-0000-1000-8000-00805f9b34fb',
    serviceControl: 'daebb240-b041-11e4-9e45-0002a5d5c51b'
  }
}
```

## Инструкция по сборке и запуску

Необходима установленная версия `node.js 22.13.0`.

**1. Запуск веб-версии (локально):**
- Установите зависимости: `npm i`
- Для запуска в dev-режиме: `npm run dev`
- Для сборки проекта: `npm run build`
- **Готовую веб-версию (desktop, mobile) можно посмотреть по ссылке: `https://jeteno-worker.github.io/`**

**2. Сборка мобильного приложения для Android (требуется Android Studio):**
- Установите зависимости: `npm i`
- Соберите проект: `npm run build`
- Добавить android config `npx cap add android`
- Синхронизируйте изменения с Android Studio: `npx cap sync`
- Откройте проект в Android Studio: `npx cap open android`
- Сделать сборку стартового проекта
- В android/build.gradle добавить:
```
allprojects {
    repositories {
        google()
        mavenCentral()
        maven { url 'https://jitpack.io' }
    }
}
```
Если уже есть этот блок кода в файле, то важно добавить туда именно `maven { url 'https://jitpack.io' }`
- В Android Studio запустите эмулятор или соберите APK-файл.
- **Готовый для тестирования APK с последней версией приложения находится в корне проекта `iqos-mvp.apk`.**

## Описание архитектуры

- Архитектура приложения представляет собой доработанную и улучшенную версию методологии **Feature-Sliced Design (FSD)**. От FSD взяты принципы слоёв, слайсов и сегментов, что обеспечивает однозначную и прозрачную направленность зависимостей в приложении, а также снижает порог входа для новых разработчиков.
- В архитектуру добавлена **модульность**, что позволяет держать связанные элементы рядом.
- Архитектурный шаблон **MVVM** отделяет модель (логику) от представления (view) и способствует написанию более гибкого кода.
- При написании менеджеров подключения к низкоуровневому API используется зависимость от интерфейсов (благодаря TypeScript), а не от конкретной реализации. Это позволяет легко подменять менеджеры подключения для web и Android, просто создав новый класс, наследующий абстрактный.

**Описание слоёв:**
*   **`app`** — содержит глобальные настройки приложения (роутинг, глобальные стили и т.п.).
*   **`assets`** — цифровые ресурсы, используемые для контента (иконки, изображения и т.п.).
*   **`shared`** — утилитарные функции и компоненты для переиспользования в модулях на любом слое.
*   **`modules`** — модули, содержащие слайсы, которыми оперирует приложение. Они делятся на слои с направленностью зависимостей сверху вниз:
    *   **`pages`** — полные страницы или крупные части страниц при вложенном роутинге.
    *   **`widgets`** — крупные самодостаточные блоки функциональности или интерфейса, обычно реализующие целый пользовательский сценарий.
    *   **`features`** — переиспользуемые реализации целых фич продукта, то есть действий, приносящих бизнес-ценность пользователю.
    *   **`entities`** — бизнес-сущности, с которыми работает проект (например, `user` или `product`).

Вышеописанные слои, в свою очередь, делятся по **сегментам**:
*   **`ui`** — всё, что связано с отображением: UI-компоненты, форматтеры, стили.
*   **`api`** — взаимодействие с бэкендом или низкоуровневым API: функции запросов, типы данных, мапперы.
*   **`model`** — модель данных: схемы валидации, интерфейсы, хранилища и бизнес-логика.
*   **`lib`** — библиотечный код, необходимый другим модулям этого слайса.
*   **`config`** — файлы конфигурации.

В данном проекте некоторые слои (`widgets`, `entities`) и сегменты (`config`, `lib`) не использовались, так как реализованный функционал этого не требовал (по сути, мы получаем данные и выводим их). Однако в более сложном приложении с обширным пользовательским опытом их применение будет необходимым и оправданным.

## Стек технологий

*   **React 19** — последняя версия UI-библиотеки для построения интерфейсов.
*   **Capacitor** — фреймворк для развёртывания веб-приложения как нативного (iOS/Android/Desktop).
*   **MobX** — менеджер состояния с реактивным программированием (store-модель).
*   **TypeScript** — статическая типизация для повышения надёжности и поддерживаемости кода.
*   **Vite** — быстрый сборщик и dev-сервер для современной веб-разработки.
*   **ESLint** — линтер для контроля качества кода и соблюдения стандартов.

#### Ключевые преимущества стека:
*   **Кроссплатформенность** (Web, iOS, Android, Desktop*).
*   Высокая производительность.
*   Типобезопасность.
*   Современный инструментарий.
*   Реактивная работа с состоянием.

_*Desktop — с использованием Electron или Tauri._

**Преимущество подхода:** ~85% единой кодовой базы для web (запуск на iOS через Bluefy по Bluetooth) и нативного Android-приложения. Основные отличия будут заключаться в методах подключения через низкоуровневое API, а также есть возможность запуска веб-версии приложения на desktop.
